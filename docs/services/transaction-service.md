# transaction-service

**Port:** 8083  
**Database:** MySQL 8.3 — `transactions_db`  
**OpenAPI spec:** [`docs/openapi/transaction-service.yaml`](../openapi/transaction-service.yaml)  
**Upstream dependencies:** account-service (via OpenFeign)

---

## Role

Records financial transactions on accounts. When a transaction is created, this service validates the account via account-service, computes the new balance, persists the transaction (including a `balance_after` snapshot), and then patches the account balance via a Feign call back to account-service.

Transactions are **immutable** once committed. There are no update or delete endpoints. A reversal is a new transaction of the opposite type.

This service also exposes `GET /api/v1/external-score/{customerId}`, which the batch-worker calls to get a derived credit score computed from transaction behaviour over the last 12 months.

## Feign Client (→ account-service)

Called when:
- `POST /api/v1/transactions` — validate account exists, get current balance, then `PATCH` updated balance after persisting

Circuit breaker configuration (in `application.yml`):

```yaml
resilience4j:
  circuitbreaker:
    instances:
      account-service:
        slidingWindowSize: 10
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
        permittedNumberOfCallsInHalfOpenState: 3
  retry:
    instances:
      account-service:
        maxAttempts: 3
        waitDuration: 500ms
  timelimiter:
    instances:
      account-service:
        timeoutDuration: 3s
```

Fallback strategy: if account-service is unreachable during transaction creation, return `503 Service Unavailable` — do not persist an unverified transaction.

## Database Schema

Flyway migration: `src/main/resources/db/migration/V1__create_transactions.sql`

```sql
CREATE TABLE transactions (
    id                  BIGINT          NOT NULL AUTO_INCREMENT,
    reference_number    VARCHAR(36)     NOT NULL,
    account_id          BIGINT          NOT NULL,
    customer_id         BIGINT          NOT NULL,
    transaction_type    ENUM('CREDIT','DEBIT','TRANSFER_IN','TRANSFER_OUT','FEE')
                                        NOT NULL,
    amount              DECIMAL(19,4)   NOT NULL,
    currency            CHAR(3)         NOT NULL DEFAULT 'EUR',
    balance_after       DECIMAL(19,4)   NOT NULL,
    description         VARCHAR(500)    DEFAULT NULL,
    transaction_date    DATETIME(3)     NOT NULL,
    created_at          DATETIME(3)     NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    PRIMARY KEY (id),
    UNIQUE KEY uq_transactions_reference    (reference_number),
    INDEX         idx_transactions_account_date  (account_id, transaction_date),
    INDEX         idx_transactions_customer_date (customer_id, transaction_date)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### Key design decisions

- `reference_number` is a UUID generated by the service — it is the idempotency key and the identifier exposed to clients. Clients can use it to look up a transaction without knowing the internal `id`.
- `customer_id` is denormalized. The batch-worker's `external-score` endpoint aggregates transaction behaviour per customer across all accounts. Without `customer_id` in this table, the query would require a cross-service join. See [banking-app-concept.md](../banking-app-concept.md) for context.
- `balance_after` is a point-in-time snapshot of the account balance immediately after this transaction. It avoids reconstructing balance history by summing all prior transactions — critical for the batch score calculation covering a 12-month window.
- `transaction_date` is the business timestamp (when the transaction occurred). `created_at` is the system audit timestamp (when the record was inserted). They can differ for backdated imports.
- Composite indexes on `(account_id, transaction_date)` and `(customer_id, transaction_date)` directly support the 12-month window queries in the batch processor.
- No `updated_at` — transactions are immutable, so an update timestamp has no meaning.

## Endpoints

Full request/response shapes are defined in the OpenAPI spec.

| Method | Path | Description | Caller |
|---|---|---|---|
| `POST` | `/api/v1/transactions` | Create transaction (validate account, patch balance) | Frontend |
| `GET` | `/api/v1/transactions` | List — paginated | Frontend |
| `GET` | `/api/v1/transactions/{id}` | Get by ID | Frontend |
| `GET` | `/api/v1/transactions/reference/{ref}` | Get by reference number (idempotency lookup) | Frontend |
| `GET` | `/api/v1/transactions/account/{accountId}` | List for account — paginated, optional date range | Frontend |
| `GET` | `/api/v1/transactions/customer/{customerId}` | List for customer across all accounts — paginated | Frontend |
| `GET` | `/api/v1/external-score/{customerId}` | Derived credit score from 12-month transaction window | batch-worker |

No `PUT`, `PATCH`, or `DELETE` — transactions are immutable by design.

### external-score calculation

The endpoint aggregates the last 12 months of transactions for a customer and returns a numeric score (0–1000) plus a letter grade (A–F). The exact scoring formula is an implementation detail of this service, but the output aligns with the batch-worker's rating classification:

| Score | Grade |
|---|---|
| 850–1000 | A |
| 700–849 | B |
| 550–699 | C |
| 400–549 | D |
| 200–399 | E |
| 0–199 | F |

The batch-worker stores both the numeric score and the grade in `customer_ratings.external_score` and uses the grade as input to the final `rating_class` calculation.
